| 问题 | 具体表现 | 解决措施 | 优先级 |

## 1️⃣ 卡顿问题 - 分析断连原因 + 技术手段解决

### 根因分析

| 可能原因 | 当前代码现状 | 影响程度 |
|---------|-------------|---------|
| **ASR 断线重连** | 有重连机制(`deepgram_asr.py`)，但重连期间可能丢帧 | 高 |
| **LLM 响应延迟** | 5s超时(`gpt4o_pipeline.py:195`)，但无中间反馈 | 高 |
| **上下文累积** | `max_short_term_tokens=2000`，对话久了 prompt 变长 | 中 |
| **TTS 首包延迟** | OpenAI TTS 首包约 300-500ms | 中 |
| **WebSocket 不稳定** | 有心跳机制，但无重连恢复 | 低 |

### 解决方案

```python
# - 3秒无响应 → 发送 "Hmm, let me think..." 填充音
```

- 在 `process_audio_stream_with_transcription` 中添加 3 秒计时器
- 超时发送预生成的填充音 (已有 `_filler_cache`)

#### 方案 C: ASR 断连平滑处理


## 2️⃣ 评价问题 - 只指出严重错误 + 多给正面反馈

### 根因分析

| 问题 | 当前代码现状 | 需要修改 |
|------|-------------|---------|
| **评分过严** | 句式复杂度上限机制过于严格 (简单句上限50分) | 放宽上限 |
| **纠错过多** | `Max 3 corrections` 但每个小错都指出 | 只纠严重错误 |
| **正面反馈不足** | `good_expressions` 往往为空 | 主动寻找亮点 |
| **评估语气** | 标准教学评估风格 | 改为成长型心态 |
| **评估不及时** | `EVALUATION_STAGE_TIMEOUT = 45s` | 降低到 20-30s |

### 解决方案

#### 方案 A: 修改评估 Prompt (核心改动)

**文件**: `prompts/templates.py`

**修改 1: 综合评估 Prompt** (`get_comprehensive_evaluation_system_prompt`)

```python
# 修改权重，降低句式权重，提高流利度/勇气权重
## 6 Dimensions & Weights (UPDATED - 鼓励型评估)
| Dimension | Weight | Focus |
|-----------|--------|-------|
| Pronunciation | 15% | 可理解即可 |
| Fluency | 25% | 敢说、愿说最重要 |  # 提高
| Grammar | 20% | 只关注影响理解的错误 |  # 降低
| Vocabulary | 15% | 有尝试就好 |
| Sentence Structure | 15% | 不过分强调复杂度 |
| Coherence | 10% | 能表达完整意思 |

# 修改句式上限机制 - 更宽松
| 句式类型 | 句式分 | 总分上限 |
|---------|--------|---------|
| 简单句 SVO | 25-35 | 最高 60 分 |  # 原50，提高
| 简单句+修饰 | 35-45 | 最高 70 分 |  # 原55，提高
| 基础复合句 | 45-55 | 最高 80 分 |  # 原60，提高
```

**修改 2: 纠错策略**

```python
## CORRECTION STRATEGY (CRITICAL - BE ENCOURAGING!)

### Only correct these "Communication-Breaking" errors:
1. **Confusing tense** that changes meaning (e.g., "I go yesterday" → unclear timeline)
2. **Wrong word** that causes misunderstanding (e.g., "I'm exciting" vs "I'm excited")
3. **Missing key structure** (e.g., "I busy" - missing verb)

### NEVER correct:
- Minor grammar that doesn't affect understanding
- Spoken English patterns (gonna, wanna, "And I was like...")
- Stylistic choices
- Capitalization/punctuation (ASR artifacts)

### ENCOURAGEMENT FIRST!
- For EVERY correction, find 2 things they did WELL
- Lead with strengths, mention improvement areas gently
- Use growth-mindset language: "You're getting there!" not "You made an error"
```

**修改 3: good_expressions 必填**

```python
## good_expressions (REQUIRED - minimum 2 items!)
- ALWAYS find at least 2 positive things, even in short responses
- Examples of what counts as "good":
  - Attempted a new word/structure
  - Used a connector (and, but, so)
  - Self-corrected
  - Spoke confidently
  - Clear pronunciation of any word
  - Completed a thought
```

#### 方案 B: 降低评估超时

**文件**: `config/constants.py`

```python

# 改: EVALUATION_STAGE_TIMEOUT = 25  # 更快反馈
```

#### 方案 C: 前端展示优化

## 3️⃣ 语音识别问题 - 测试上下文长度与延时关系 + 优化

### 根因分析

| 问题 | 当前代码现状 | 影响程度 |
|------|-------------|---------|
| **ASR 识别不准** | Deepgram Nova-2 通用模型 | 中 |
| **英语学习者口音** | 无特殊优化 | 中 |
| **噪音环境** | 无降噪处理 | 低 |
| **对话累积延迟** | history 不断增长，LLM prompt 变长 | 高 |
| **System Prompt 过长** | 包含大量指令 | 中 |

### 解决方案

#### 方案 A: 验证上下文长度 vs 延迟关系

**测试方法**:
```python
# 记录每轮对话:
# 1. conversation_history 的 token 数
# 2. LLM TTFT (首 token 时间)
# 3. 总延迟

# 预期发现: token 数 ↑ → 延迟 ↑
# 解决: 更积极的截断/摘要
```

#### 方案 B: 更激进的记忆压缩

**文件**: `models/conversation_memory.py`

**文件**: `services/deepgram_asr.py`


#### 方案 D: System Prompt 瘦身

## 🔬 验证方法

### 评价问题验证
1. 用户说 "I like eat apple" → 分数应 ≥ 40 (非 30)
2. 短句如 "Yes I do" → 不应指出任何错误
3. 每条评估必须有 ≥ 2 条 good_expressions

### 卡顿问题验证
1. 监控 `total_latency_ms` 平均值 < 2000ms
2. 对话 10 轮后延迟不应比第 1 轮增加 > 50%
3. 用户无法感知 ASR 重连

### 语音识别问题验证
1. 绘制 token 数 vs 延迟曲线
2. 压缩后延迟降低 > 20%
3. ASR 词错误率 (WER) 保持或降低
