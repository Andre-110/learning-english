## 三、代码层面问题检测

### 1. 延时问题分析

**相关代码位置**：
- `api/openrouter_audio_endpoint.py` 第 520-570 行（交互轨处理）
- `api/openrouter_audio_endpoint.py` 第 620-625 行（评估轨等待）

**发现的问题**：
- [ ] 交互轨超时设置为 30 秒（第 527 行）
- [ ] 评估轨超时设置为 60 秒（第 622 行）
- [ ] 无重试机制，超时直接失败
- [ ] 无心跳检测，长时间等待用户无感知

**代码证据**：
```python
# 第 527 行 - 交互轨 30 秒超时
msg_type, data = await loop.run_in_executor(
    None, lambda: chunk_queue.get(timeout=30)
)

# 第 622 行 - 评估轨 60 秒超时
status, result, eval_time = await loop.run_in_executor(
    None, lambda: eval_result_queue.get(timeout=60)
)
```

### 2. 卡顿问题分析

**相关代码位置**：
- `api/openrouter_audio_endpoint.py` 第 778-785 行（TTS 音频分块发送）

**发现的问题**：
- [ ] TTS 音频分块大小为 16KB，每块间隔 3ms
- [ ] S2S 音频直接转发，无缓冲机制
- [ ] 前端可能存在音频播放不连续问题

**代码证据**：
```python
# 第 778-785 行 - TTS 分块发送
chunk_size = 16 * 1024  # 16KB
for i in range(0, len(audio_data), chunk_size):
    chunk = audio_data[i:i + chunk_size]
    await websocket.send_json({
        "type": "audio_chunk",
        "data": base64.b64encode(chunk).decode()
    })
    await asyncio.sleep(0.003)  # 3ms 间隔
```

### 3. 纠错误判问题分析

**相关代码位置**：
- `prompts/templates.py` 评估轨 Prompt

**发现的问题**：
- [ ] 转录可能被 S2S 模型自动翻译（如"土豆"→"potato"）
- [ ] 评估轨可能对正确表达过度纠正
- [ ] Prompt 中的规则可能未被模型严格遵守

### 4. 交互繁琐问题分析

**相关代码位置**：
- `frontend/src/stores/conversation.js`（前端录音控制）

**发现的问题**：
- [ ] 需要手动点击开始/结束录音
- [ ] 缺少语音活动检测（VAD）自动开始/结束
- [ ] 空格键快捷键可能不够直观

## 四、改进建议

### 立即执行（P0）

1. **优化延时**
   - 添加处理进度心跳（每 5 秒发送一次）
   - 降低超时时间，添加重试机制
   - 考虑使用更快的模型或 API

2. **解决卡顿**
   - 前端实现音频预缓冲（至少缓冲 2-3 个 chunk 再播放）
   - 增大 TTS 分块大小或减少网络往返
   - 考虑使用流式 TTS 而非全量生成后分块

### 短期执行（P1）

3. **提升纠错准确性**
   - 优化转录 Prompt，强调保留原始语言
   - 添加"只纠正真正错误"的规则
   - 考虑使用专门的 ASR 模型做转录

4. **简化交互**
   - 实现 VAD（语音活动检测）自动开始/结束
   - 添加长按录音模式
   - 优化空格键交互提示

### 中期执行（P2）

5. **评分校准**
   - 收集更多用户数据校准评分标准
   - 添加评分解释说明

6. **增强个性化**
   - 显示用户画像和学习进度
   - 根据用户水平动态调整难度
