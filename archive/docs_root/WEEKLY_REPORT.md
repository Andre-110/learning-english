## 二、技术方案对比

### 2.1 延迟优化方案演进

| 方案 | 流程 | 延迟 | 状态 | 选择原因 |
|------|------|------|------|----------|
| 标准流程 | STT→LLM×2→TTS | 8-11s | ❌ 废弃 | 延迟过高 |
| OpenRouter Audio | Audio→GPT-4o→TTS | 4.8s | ⚠️ 备选 | 成本高，偶有429错误 |
| **Qwen3-Omni** | Audio→Qwen-Omni→TTS | **3-5s** | ✅ 当前 | 延迟低，成本低，稳定 |
| Realtime API | Audio→GPT-4o-realtime→Audio | ~500ms | ❌ 不可用 | yunwu.ai 不支持 WebSocket |

### 2.2 为什么选择 Qwen3-Omni

1. **延迟最优**：一次调用完成转录+评估+生成，省去 STT 和多次 LLM 调用
2. **成本最低**：阿里云 DashScope 价格远低于 OpenAI
3. **稳定性好**：无 429 限流问题
4. **中文支持**：对中英混合场景处理更好

## 七、经验总结

### 7.1 Prompt 工程是核心

本次优化中，**80% 的问题通过修改 Prompt 解决**：
- 话题跟随 → System Prompt 增加优先级规则
- 评分校准 → 明确 CEFR 分数区间和高级表达示例
- 情感连接 → 提供正反例，禁止公式化回复
- 多语言处理 → 区分"纠错"和"翻译教学"

### 7.2 端到端模型是趋势

从 STT+LLM+TTS 分离架构迁移到 Qwen3-Omni 端到端模型：
- 延迟从 60s+ 降到 3-5s
- 代码复杂度降低
- 成本降低

### 7.3 用户反馈驱动迭代

- 2位用户的真实反馈比100条假设更有价值
- "AI不跟随话题"这个核心问题在内部测试中未发现
- 建议：每周至少1次真实用户测试
