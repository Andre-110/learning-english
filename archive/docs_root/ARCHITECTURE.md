# 系统架构设计文档

## 一、架构设计原则

### 1. 高内聚低耦合
- **模块内聚性**：每个模块职责单一明确
- **模块耦合度**：通过接口抽象实现松耦合，依赖注入降低依赖

### 2. 分层架构
```
┌─────────────────────────────────────┐
│         API层 (api/)               │  ← HTTP接口，请求处理
├─────────────────────────────────────┤
│     核心业务层 (core/)              │  ← 业务逻辑编排
├─────────────────────────────────────┤
│      服务层 (services/)             │  ← 具体服务实现
├─────────────────────────────────────┤
│    提示词层 (prompts/)              │  ← 提示词管理
├─────────────────────────────────────┤
│      数据模型层 (models/)           │  ← 数据结构定义
├─────────────────────────────────────┤
│      存储层 (storage/)              │  ← 数据持久化
└─────────────────────────────────────┘
```

## 二、模块职责划分

### 1. API层 (`api/`)
**职责**：
- 接收HTTP请求
- 参数验证
- 调用业务层服务
- 返回HTTP响应

**关键文件**：
- `api/main.py`: FastAPI应用入口，路由定义

**设计特点**：
- 使用依赖注入获取服务实例
- Pydantic模型进行请求/响应验证
- 统一的错误处理

### 2. 核心业务层 (`core/`)
**职责**：
- 编排业务流程
- 实现业务规则
- 协调各服务

**关键模块**：
- `ConversationManager`: 管理对话生命周期
- `EvaluationEngine`: 协调评估流程
- `AdaptationEngine`: 实现自适应逻辑

**设计特点**：
- 不直接依赖具体实现，依赖抽象接口
- 单一职责原则
- 易于测试和扩展

### 3. 服务层 (`services/`)
**职责**：
- 提供具体业务服务
- 封装LLM调用
- 实现评估、生成、上下文管理等核心功能

**关键服务**：
- `LLMService`: 统一的LLM调用接口（支持多提供商）
- `EvaluatorService`: 执行评估任务
- `QuestionGeneratorService`: 生成适配题目
- `ContextManagerService`: 管理对话上下文和摘要

**设计特点**：
- 策略模式：LLM提供商可插拔
- 工厂模式：LLM服务创建
- 接口隔离：细粒度服务接口

### 4. 提示词层 (`prompts/`)
**职责**：
- 管理提示词模板
- 动态组装提示词
- 支持版本管理

**关键模块**：
- `templates.py`: 提示词模板定义
- `builders.py`: 提示词构建器

**设计特点**：
- 模板化设计，易于修改和优化
- 支持动态参数注入
- 模块化提示词组件

### 5. 数据模型层 (`models/`)
**职责**：
- 定义核心数据结构
- 数据验证
- 业务对象建模

**关键模型**：
- `UserProfile`: 用户能力画像
- `Conversation`: 对话会话
- `AssessmentResult`: 评估结果

**设计特点**：
- 使用Pydantic进行数据验证
- 清晰的领域模型
- 类型安全

### 6. 存储层 (`storage/`)
**职责**：
- 数据持久化抽象
- 提供数据访问接口
- 支持多种存储后端

**关键模块**：
- `repository.py`: 存储接口定义
- `impl/memory_repository.py`: 内存存储实现

**设计特点**：
- 仓储模式（Repository Pattern）
- 接口与实现分离
- 易于扩展（可添加数据库、Redis等实现）

## 三、依赖关系图

```
API层
  ↓ (依赖)
核心业务层
  ↓ (依赖)
服务层 ← → 提示词层
  ↓ (依赖)
数据模型层
  ↑ (使用)
存储层
```

## 四、关键设计模式

### 1. 依赖注入（Dependency Injection）
- API层通过`Depends`注入服务
- 服务层通过构造函数注入依赖
- 降低耦合，提高可测试性

### 2. 策略模式（Strategy Pattern）
- LLM提供商可插拔（OpenAI、Anthropic）
- 存储后端可切换（内存、数据库、Redis）

### 3. 工厂模式（Factory Pattern）
- `LLMServiceFactory`: 创建LLM服务实例
- `RepositoryFactory`: 创建存储实例

### 4. 仓储模式（Repository Pattern）
- 抽象数据访问层
- 业务逻辑与数据存储解耦

## 五、数据流

### 对话流程
```
用户请求 → API层
         ↓
     ConversationManager
         ↓
    ┌────┴────┐
    ↓         ↓
Evaluator  Generator
    ↓         ↓
    └────┬────┘
         ↓
    更新用户画像
         ↓
    返回响应
```

### 评估流程
```
用户回答 → EvaluatorService
         ↓
    构建评估提示词
         ↓
    LLMService调用
         ↓
    解析评估结果
         ↓
    更新能力画像
```

## 六、扩展点

### 1. 新增LLM提供商
- 实现`LLMService`接口
- 在`LLMServiceFactory`中注册

### 2. 新增存储后端
- 实现`ConversationRepository`和`UserRepository`接口
- 在`RepositoryFactory`中注册

### 3. 新增评估维度
- 修改`EvaluationPrompt`模板
- 更新`AssessmentResult`模型

### 4. 优化提示词
- 修改`prompts/templates.py`中的模板
- 支持A/B测试（可扩展版本管理）

## 七、性能优化策略

### 1. 上下文管理
- 动态摘要：定期压缩对话历史
- 减少token消耗

### 2. 缓存机制
- 题目模板缓存（可扩展）
- 用户画像缓存

### 3. 模型选择
- 高性能模型用于评估和生成
- 轻量模型用于摘要等辅助任务

## 八、测试策略

### 1. 单元测试
- 各服务独立测试
- Mock依赖

### 2. 集成测试
- API端点测试
- 完整流程测试

### 3. 提示词测试
- 评估一致性测试
- 输出格式验证
